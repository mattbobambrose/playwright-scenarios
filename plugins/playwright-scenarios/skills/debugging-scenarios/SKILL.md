# Debugging Scenarios

When tests generated by `/scenario-to-tests` fail, this skill guides systematic troubleshooting. Most failures fall into a small number of root causes.

## Diagnosis order

Work through these in order. Each cause has a detection method and a fix. Stop at the first match — multiple failures usually share one root cause.

### 1. Iframe not declared

**Symptoms:** Every selector in the test fails with "element not found" or "timeout waiting for selector." The page loads successfully (no 404/500), but nothing inside it can be located.

**Detection:** Check the scenario file for an `**Iframe:**` tag. If absent, open the URL with `playwright-cli` and run `playwright-cli snapshot` — look for `<iframe>` elements in the snapshot. If the form/content lives inside an iframe, the scenario is missing the tag.

**Fix:** Add `**Iframe:** <selector>` at the top of the scenario. Use the iframe's `id` (`#stripe-payment-iframe`), `name` attribute (`[name="checkout"]`), or `src` pattern (`iframe[src*="stripe.com"]`). Re-run `/scenario-to-tests`.

### 2. Selector drift

**Symptoms:** Some tests pass, others fail with "element not found" on specific selectors. The failures reference elements that used to exist but have been renamed, moved, or restructured.

**Detection:** Open the URL with `playwright-cli`, navigate to the failing test's context, and run `playwright-cli snapshot`. Compare the snapshot's element text/roles/labels against the scenario's Action/Expected bullets. Look for:
- Button/link text changes ("Log In" → "Sign In").
- Label changes ("Email" → "Email address").
- Element removal or restructuring.

**Fix:** Update the scenario's Action/Expected bullets to match the current DOM. Then re-run `/review-scenario <name>` to verify all claims, and `/scenario-to-tests <name>` to regenerate.

### 3. Timing / race condition

**Symptoms:** Tests pass inconsistently — sometimes pass, sometimes fail on the same assertion. Failures often mention "timeout" and target elements that exist but aren't visible yet.

**Detection:** The element exists in the DOM but isn't interactable due to:
- An animation or transition in progress.
- A loading spinner covering the element.
- An API call that hasn't returned yet.
- A lazy-loaded section that hasn't rendered.

**Fix options (in preference order):**
1. If a specific element signals "ready," add an Expected assertion before the failing Action: `- **Expected:** The loading spinner is not visible`.
2. If the page has a loader, add `**Assert throughout:** No loading spinners visible after navigation` at the scenario top.
3. As a last resort, add `**Timeout:** 60000` to the scenario or the specific test to increase the wait. This is a band-aid — prefer fixing the race.

### 4. Cross-origin / navigation boundary

**Symptoms:** Tests fail after a navigation that changes the page's origin (e.g., redirected to an OAuth provider, a payment processor, or a different subdomain). Selectors stop matching after the navigation.

**Detection:** Check the URL in the scenario's flow. If it transitions from `app.example.com` to `auth.provider.com` or from an iframe to the top-level page, there's a boundary.

**Fix:**
- For iframe-to-page transitions: add `**Iframe:** none` at the point where actions return to the top-level page.
- For cross-origin redirects: the scenario may need to assert the redirect URL (`- **Expected:** The URL changes to https://auth.provider.com/...`) and then resume after the redirect. Playwright handles same-tab navigations automatically, but cross-origin popups need `context.waitForPage()` in the test code — flag this as needing manual test adjustment.

### 5. Stale fixture data

**Symptoms:** Tests fail because entered data is rejected by the form (validation errors, "email already taken," "invalid phone format").

**Detection:** Check the fixture file or inline data bullets. Common causes:
- Email addresses that already exist in the system (tests created an account in a prior run).
- Phone numbers in wrong format (parentheses vs. raw digits).
- Dates that have expired.
- Values that don't match the form's acceptable options.

**Fix:**
- For unique-per-run fields (email), use a template in the fixture: `"email_prefix": "sarah-test"` and have the test append a timestamp: `sarah-test-1713200000@example.com`.
- For format issues, update the fixture to match what the form actually accepts (raw digits, ISO dates, etc.).
- For stale option values, re-run `/review-scenario` to check the form's current options.

### 6. Generated code structure wrong

**Symptoms:** Tests fail to compile or the test runner can't find the class. Errors like "unresolved reference," "class not found," or "wrong package."

**Detection:** Check:
- The scenario-name → class-name conversion (see `/scenario-to-tests` conversion rules). Does the filename match?
- The `SCENARIOS_PACKAGE` derivation — does the package declaration match the directory structure?
- The `BASE_TEST_CLASS` — does the test extend the right base class? Does that class exist?

**Fix:** Run `/playwright-scenarios-config` to verify `test_dir`, `source_root`, and `base_test_class` are correct. If `source_root` was inferred wrong, set it explicitly. Re-run `/scenario-to-tests`.

## After diagnosis

Once the root cause is identified and the scenario or fixture is fixed:

1. Re-run `/review-scenario <name>` to verify the fix against the live site.
2. Re-run `/scenario-to-tests <name>` to regenerate the test code.
3. Run the tests: `./gradlew test --tests "<package>.<ClassName>"` to verify the fix.

If the test still fails after all 6 checks, the issue is likely in the test generator's output rather than the scenario. Read the generated test file directly and compare it against the scenario — look for translation errors (wrong locator strategy, missing await, incorrect assertion method).

## What this skill does NOT do

- It does not auto-fix scenarios — it guides the user to the root cause and suggests the fix.
- It does not run tests — that's `/scenario-to-tests` Phase 4.
- It does not open a browser — that's `/review-scenario` Phase 2. (But it may suggest the user run `/review-scenario` as a diagnostic step.)
